## Key beats

### 1. Start with the use case

Many systems operate over lists — leads, tickets, candidates, companies, docs, markets. Tables give those records a persistent, inspectable home.

### 2. Define a table

- Rows = records / items.
- Columns = fields / properties.
- Column types: text, number, boolean, date, JSON.
- Framing: spreadsheet-like visually, but really closer to **structured state** / a datatable.

### 3. What belongs in tables

Statuses, scores, categories, structured extraction results, research targets, workflow outputs — anything you want to filter, update, compare, or process repeatedly.

### 4. Connect to workflows

Workflows can read, write, and update tables, and tables make workflow outputs visible.

> Workflows do the work. Tables keep the work inspectable.

### 5. Preview active tables

A workflow can run across rows and write outputs back into columns. Don't deep-dive yet — just plant the idea.

> Instead of moving data into a pipeline, the pipeline can come to the data.

## Avoid

- Don't teach workflow columns deeply here.
- Don't lean too hard on the spreadsheet comparison.
- Don't dwell on other tools — at most, a brief "similar row-wise AI work pattern."
