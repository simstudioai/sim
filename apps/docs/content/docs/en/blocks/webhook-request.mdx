The Webhook (outbound) block sends HTTP POST requests to external webhook endpoints. It supports custom headers, automatic HMAC-SHA256 payload signing for secure delivery, and AI-assisted JSON generation.

## Configuration Options

### URL

The webhook endpoint URL where the POST request will be sent.

- **Required**: Yes
- **Type**: String (full URL, starting with http:// or https://)
- **Example**: `https://api.example.com/webhooks/events`

### Body

The JSON payload to send in the POST request body.

- **Type**: JSON object
- **AI Wand**: Use the AI wand to automatically generate JSON based on a natural language description
- **Example**:
```json
{
  "event": "order_created",
  "orderId": "<api.order_id>",
  "amount": 99.99,
  "timestamp": "<function.currentTime>"
}
```

### Secret (Optional)

A shared secret used to sign the payload with HMAC-SHA256. If provided, the webhook request will include:

- **Header**: `X-Signature` containing the HMAC-SHA256 hash of the payload
- **Purpose**: Verify the request authenticity on the receiving end

**How to validate on the receiver:**
```javascript
const crypto = require('crypto')
const signature = crypto
  .createHmac('sha256', secret)
  .update(JSON.stringify(body))
  .digest('hex')
const isValid = signature === req.headers['x-signature']
```

### Headers (Optional)

Custom HTTP headers to include with the request. Specify as key-value pairs:

- **Content-Type**: Defaults to `application/json`
- **Custom Headers**: Add authorization, tracking IDs, API keys, or other headers as needed

**Example:**
```
Authorization: Bearer token123
X-Custom-Header: custom-value
X-Request-ID: <function.requestId>
```

## Outputs

- **`<webhookRequest.data>`**: The response body from the webhook endpoint (JSON)
- **`<webhookRequest.status>`**: HTTP status code of the response (e.g., 200, 404, 500)
- **`<webhookRequest.headers>`**: Response headers as a JSON object

## Example Use Cases

**Real-time Event Notifications**
```
Agent (Process Event) → Webhook Request (Notify external system) → Response
```

**Secure Third-party Integrations**
```
Function (Generate payload) → Webhook Request (w/ HMAC signature) → External Service
```

**Multi-service Orchestration**
```
API (Fetch data) → Function (Transform) → Webhook Request (Service A)
                                        → Webhook Request (Service B)
```

**Audit Trail and Logging**
```
Agent → Function (Log event) → Webhook Request (Send to audit service) → Response
```

## Best Practices

- **Use HMAC signing for sensitive data**: Always enable the secret/signature option when sending sensitive information to ensure authenticity
- **Handle retries**: The receiving service should implement idempotency checks to handle duplicate webhook deliveries
- **Include correlation IDs**: Add unique request IDs to track webhook events end-to-end
- **Validate webhook responses**: Check the status code and response data before proceeding in downstream blocks
- **Use the AI wand for JSON**: Let the AI wand generate complex JSON payloads to save time and reduce errors

<Callout type="info">
  Webhook requests are synchronous—the block waits for the receiving service to respond before continuing. For better reliability with external services, implement timeout handling and consider retry logic in your workflow.
</Callout>

## Webhook Security

When sending sensitive data via webhooks:

1. **Enable HMAC signing**: Always use the Secret field to cryptographically sign payloads
2. **Use HTTPS**: Ensure the webhook URL uses HTTPS, not HTTP
3. **Validate on receiver**: The receiving service should validate the signature before processing
4. **Rotate secrets periodically**: Change secrets regularly to minimize impact of compromise
5. **Log requests**: Keep audit logs of all webhook requests for compliance
