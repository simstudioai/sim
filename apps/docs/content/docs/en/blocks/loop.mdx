The **Loop block** is a container: you drop other blocks inside it, and it runs them again and again. Use it to walk through a collection one item at a time, repeat an operation a set number of times, or keep going until a condition flips. Each pass runs the contained blocks in order, and the loop collects what each pass produces into a single array you can read once it finishes.

Iterations run **one after another**, each waiting for the last to complete. That ordering is what makes loops the right choice when a pass depends on the one before it. When the passes are independent and you'd rather run them at the same time, reach for a [Parallel](/blocks/parallel) block instead.

## Loop Types

The loop type decides how many times the block runs and what each pass has access to. Pick it on the block, then set the one value that type needs.

Use it when you want to repeat an operation a known number of times rather than walk a collection, like generating five variations of a draft.

## Configuration

Beyond the loop type, each type needs one value to control how it runs:

- **Iterations** — the number of times to run, for a For loop.
- **Collection** — the array or object to walk, for a ForEach loop. This is usually a reference to an earlier output, like `<api.data>`.
- **Condition** — a boolean expression checked each pass, for While and Do-While loops. Reference a workflow variable or block output, such as `<variable.retryCount> < 5`.

## Referencing loop data

How you reference the loop's data depends on where you are — inside the container, you read the current pass; outside, after it finishes, you read the collected results. These are two different things, so the syntax is different too.

<Tabs items={['Inside the loop', 'After the loop']}>
  <Tab>
    Blocks placed inside the loop can read the current pass with the `<loop.*>` references:

    - `<loop.index>` — the current iteration number, starting at 0.
    - `<loop.currentItem>` — the item being processed this pass (ForEach only).
    - `<loop.items>` — the full collection being walked (ForEach only).

    ```javascript
    const idx = <loop.index>;        // 0, 1, 2, ...
    const item = <loop.currentItem>; // the current item
    ```

    <Callout type="info">
      These references only resolve for blocks inside the loop container. A block outside the loop can't see them.
    </Callout>
  </Tab>
  <Tab>
    Once the loop finishes, the results from every pass are collected into an array. Read it from a block after the loop using the loop block's **name** — not `<loop.*>`, which only works inside:

    ```javascript
    // for a loop named "Process Items"
    const all = <processitems.results>; // [result1, result2, ...]
    ```

    <Callout type="info">
      The name is normalized to lowercase with spaces removed, so "Process Items" becomes `processitems`. The results stay in iteration order.
    </Callout>
  </Tab>
</Tabs>

## Nesting and limits

Containers can nest. You can place a loop inside another loop, a [Parallel](/blocks/parallel) inside a loop, or any other combination, to build multi-dimensional workflows — for example, looping over accounts and, for each one, looping over that account's orders.

For and ForEach loops are capped at **1,000 iterations** to keep runs bounded. While and Do-While loops have no hard cap — they run until their condition becomes false — so make sure the condition will eventually flip, or the loop won't end.

## Best Practices

- **Use ForEach for collections.** When you have an array or object to process, ForEach reads more clearly than a For loop where you index into the collection by hand.
- **Keep iteration counts reasonable.** Because passes run one after another, every extra iteration adds to the total run time. If you're processing thousands of items, consider batching the work.
- **Reach for Parallel when order doesn't matter.** Independent work over a collection finishes far faster in a [Parallel](/blocks/parallel) block, since the passes run concurrently instead of in sequence.
- **Guarantee an exit for condition loops.** For While and Do-While loops, double-check that something inside the loop moves the condition toward false — usually a [Variables](/blocks/variables) block updating a counter — so the loop is guaranteed to stop.
