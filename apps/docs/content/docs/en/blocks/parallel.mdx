The **Parallel block** is a container that runs the blocks inside it many times at once — once per item in a collection, or a fixed number of times. It's the concurrent counterpart to a [Loop](/blocks/loop): the work is split across instances that run at the same time, and their results are collected into one array. Use it when the passes are independent and order doesn't matter; when each pass depends on the previous one, use a Loop instead.

## Parallel Types

The type decides how the work is split. Pick it on the block, then set the value it needs.

## Configuration

Beyond the type, set the value that controls how the work is split:

- **Count** — how many instances to run, for a count-based parallel.
- **Collection** — the array or object to distribute, for a collection-based parallel. This is usually an earlier output, like `<api.data>`.
- **Batch size** — how many instances run at once, from 1 to 20 (default 20). Lower it to stay under a rate-limited API.

## Referencing parallel data

As with a loop, references differ depending on where you are — inside an instance you read its own context, and after the block finishes you read the collected results.

<Tabs items={['Inside the parallel', 'After the parallel']}>
  <Tab>
    Blocks inside the parallel read the current instance with the `<parallel.*>` references:

    - `<parallel.index>` — this instance's number, starting at 0.
    - `<parallel.currentItem>` — the item assigned to this instance (collection-based only).
    - `<parallel.items>` — the full collection being distributed (collection-based only).

    <Callout type="info">
      These only resolve inside the parallel container. Each instance runs in isolation, so it can't read another instance's values.
    </Callout>
  </Tab>
  <Tab>
    After the block finishes, the results from every instance are gathered into an array. Read it from a later block using the parallel block's **name** — not `<parallel.*>`, which only works inside:

    ```javascript
    // for a parallel named "Process Tasks"
    const all = <processtasks.results>; // [result1, result2, ...]
    ```

    For large result sets, reference just the entry or field you need, like `<processtasks.results[10][0].id>`. Sim keeps results indexable and only hydrates an oversized entry when that specific path is referenced.
  </Tab>
</Tabs>

## Parallel vs Loop

Both containers repeat the blocks inside them; the difference is whether the passes run together or in sequence.

| | Parallel | Loop |
| --- | --- | --- |
| **Execution** | Concurrent | Sequential |
| **Order** | Not guaranteed | Preserved |
| **Best for** | Independent work | Dependent steps |
| **Resource use** | Higher | Lower |

## Nesting and limits

Containers can nest — a parallel inside a parallel, a loop inside a parallel, or any combination — to build multi-dimensional workflows, like fanning out over accounts and, within each, over that account's records.

A Parallel runs up to **20 instances at a time**; larger counts or collections run in further batches once the current one finishes. Because instances run concurrently, keep an eye on API rate limits and memory use when the dataset is large.

## Best Practices

- **Use it for independent work only.** Instances can't share state, so each one has to stand on its own. If a pass needs the result of the previous pass, use a [Loop](/blocks/loop).
- **Mind rate limits.** Concurrency multiplies your request rate, so lower the batch size or add a [Wait](/blocks/wait) for API-heavy work.
- **Handle errors per instance.** A failure in one instance doesn't stop the others, so each instance should handle its own errors rather than relying on a shared fallback.
- **Add an identifier if order matters.** Results aren't guaranteed to come back in input order, so include an index or id in each instance's output if you need to line them up afterward.
