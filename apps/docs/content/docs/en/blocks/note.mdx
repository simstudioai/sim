The Note block adds contextual annotations directly onto your workflow canvas. Use it to document decisions, share instructions, or leave collaborative context without affecting workflow execution.

<Callout type="info">
  Note blocks are purely for documentation and collaboration—they do not execute during workflow runs and do not affect workflow logic or output.
</Callout>

## Configuration Options

### Content

The note content displayed on the canvas. Notes support full Markdown rendering, including:

- **Text formatting**: Bold, italics, links, code inline
- **Lists**: Bullet points and numbered lists
- **Code blocks**: Syntax-highlighted code samples
- **Headings**: Multiple heading levels for organization
- **YouTube embeds**: Embed videos directly using standard embed syntax
- **Block references**: Link to other blocks in the workflow using block names

### Example Markdown

```markdown
## Processing Strategy

This workflow handles customer support tickets using:
1. Route by intent (FAQ vs. escalation)
2. Generate response with Agent
3. Log to database for analytics

**Important**: Always include ticket ID in response.

[More info](https://docs.example.com)
```

## Use Cases

**Workflow Decision Documentation**
```
Add a Note explaining why specific routing logic is used, business rules, or decision points that other team members should understand
```

**Instructions for Collaborators**
```
Leave notes with setup instructions, configuration requirements, or prerequisites before certain blocks
```

**Embedded Learning Resources**
```
Include links to documentation, YouTube tutorials, or examples that help new team members understand the workflow
```

## Best Practices

- **Keep notes concise**: Use notes to supplement, not replace, clear workflow design. If you're writing extensive documentation, consider external resources
- **Update with workflow changes**: When you modify workflow logic, update related notes to reflect the changes
- **Use for team communication**: Notes are visible to all collaborators—use them to explain non-obvious design decisions
- **Link to resources**: Include links to external documentation, FAQs, or decision logs for more context
