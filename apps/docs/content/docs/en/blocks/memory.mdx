The Memory block stores and retrieves conversation memories for AI agents. It allows you to add, retrieve, update, and delete conversation history entries, enabling stateful multi-turn interactions beyond the Agent block's built-in memory modes.

<Callout type="warning">
  Do not use the Memory block unless explicitly needed. For natural conversations, use the Agent block's built-in memory modes (Conversation, Sliding Window by messages, or Sliding Window by tokens). The Memory block is intended for injecting artificial memory into workflows when the Agent's native memory handling is insufficient.
</Callout>

## Configuration Options

### Operation

Select the memory operation to perform:

- **Add Memory**: Add a new message to the conversation memory
- **Get Memory**: Retrieve a specific memory entry by its ID
- **Get All Memories**: Retrieve all memories for a conversation
- **Delete Memory**: Remove a memory entry from storage

### Conversation ID

The unique identifier for the conversation. This key persists the memory across workflow runs and enables context sharing between multiple executions.

- **Required**: Yes
- **Type**: String
- **Example**: `chat-session-123`, `user-456`, `conversation-abc`

### Role (Add Memory only)

Specifies the speaker role for the memory entry when using the Add Memory operation:

- **user**: Message from the end user
- **assistant**: Message from the AI model
- **system**: System-level instruction or context

### Content (Add Memory only)

The message content to store when using the Add Memory operation:

- **Type**: String (Markdown supported)
- **Max Length**: No hard limit, but keep reasonably sized for performance

## Outputs

- **`<memory.memories>`**: JSON array of all memories in the conversation (for Get All Memories)
- **`<memory.memory>`**: The specific memory entry retrieved (for Get Memory)
- **`<memory.success>`**: Boolean indicating whether the operation completed successfully
- **`<memory.id>`**: The conversation ID used for the operation

## Example Use Cases

**Multi-Turn Conversation with Context Preservation**
```
User Input → Memory (Get All) → Agent (w/ history) → Memory (Add user msg) → Agent → Memory (Add assistant msg) → Response
```

**Building Persistent User Profiles**
```
API (User Data) → Memory (Add) → Agent (Enhanced w/ user context) → API (Update Profile)
```

**Conversation Analytics**
```
Trigger (Scheduled) → Memory (Get All, conversationId) → Function (Analyze sentiment) → API (Log analytics)
```

## Best Practices

- **Use Agent memory first**: The Agent block's built-in memory modes handle most conversation scenarios without requiring the Memory block
- **Manage memory growth**: Periodically clean up old memories to prevent excessive storage and slow retrieval
- **Consistent conversation IDs**: Use meaningful, consistent IDs to ensure memories are properly grouped and retrievable
- **Combine with Agent block**: For best results, use Memory alongside the Agent block to build full conversation context
