メモリブロックは、AIエージェント用の会話メモリを保存および取得します。会話履歴エントリを追加、取得、更新、削除することができ、エージェントブロックの組み込みメモリモード以上のステートフルなマルチターン相互作用を実現します。

<Callout type="warning">
  メモリブロックは明示的に必要でない限り、使用しないでください。自然な会話については、エージェントブロックの組み込みメモリモード（会話、メッセージごとのスライディングウィンドウ、トークンごとのスライディングウィンドウ）を使用してください。メモリブロックは、エージェントのネイティブメモリ処理が不十分な場合に、ワークフローに人工メモリを注入するために意図されています。
</Callout>

## 構成オプション

### 操作

実行するメモリ操作を選択します：

- **メモリを追加**：会話メモリに新しいメッセージを追加
- **メモリを取得**：IDでメモリエントリを取得
- **すべてのメモリを取得**：会話用にすべてのメモリを取得
- **メモリを削除**：ストレージからメモリエントリを削除

### 会話ID

会話の一意の識別子。このキーはワークフロー実行全体でメモリを保持し、複数の実行間でコンテキスト共有を有効にします。

- **必須**：はい
- **タイプ**：文字列
- **例**：`chat-session-123`、`user-456`、`conversation-abc`

### 役割（メモリを追加する場合のみ）

メモリを追加操作を使用するときのメモリエントリの話者役割を指定します：

- **user**：エンドユーザーからのメッセージ
- **assistant**：AIモデルからのメッセージ
- **system**：システムレベルの指示またはコンテキスト

### コンテンツ（メモリを追加する場合のみ）

メモリを追加操作を使用するときに保存するメッセージコンテンツ：

- **タイプ**：文字列（Markdown対応）
- **最大長**：厳密な制限はありませんが、パフォーマンスのため妥当なサイズを保つでください

## 出力

- **`<memory.memories>`**：会話内のすべてのメモリのJSON配列（すべてのメモリを取得の場合）
- **`<memory.memory>`**：取得したメモリエントリ（メモリを取得の場合）
- **`<memory.success>`**：操作が正常に完了したかどうかを示すブール値
- **`<memory.id>`**：操作に使用した会話ID

## ユースケース例

**コンテキスト保持を備えたマルチターン会話**
```
ユーザー入力 → メモリ（すべてを取得） → エージェント（履歴付き） → メモリ（ユーザーメッセージを追加） → エージェント → メモリ（アシスタントメッセージを追加） → レスポンス
```

**永続的なユーザープロファイルの構築**
```
API（ユーザーデータ） → メモリ（追加） → エージェント（ユーザーコンテキスト強化） → API（プロファイル更新）
```

**会話分析**
```
トリガー（スケジュール） → メモリ（すべてを取得、会話ID） → ファンクション（感情分析） → API（分析ログ）
```

## ベストプラクティス

- **エージェントメモリを最初に使用**：エージェントブロックの組み込みメモリモードは、メモリブロックを必要とせず、ほとんどの会話シナリオを処理します
- **メモリ増加を管理**：古いメモリを定期的にクリーンアップして、過度なストレージと遅い取得を防止します
- **一貫した会話ID**：意味のある一貫したIDを使用して、メモリが適切にグループ化され、取得可能であることを確認します
- **エージェントブロックと組み合わせ**：最良の結果のために、エージェントブロックと一緒にメモリを使用して、完全な会話コンテキストを構築します
