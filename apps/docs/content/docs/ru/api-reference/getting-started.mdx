## Базовый URL

Все запросы к API выполняются по адресу:

```
https://www.sim.ai
```

## Быстрый старт

<Steps>

<Step>
### Получите ваш API-ключ

Перейдите на платформу Sim, откройте **Settings**, затем перейдите в **Sim Keys** и нажмите **Create**. Подробнее о типах ключей см. в разделе [Аутентификация](/api-reference/authentication).
</Step>

<Step>
### Найдите ID вашего рабочего процесса

Откройте рабочий процесс в редакторе Sim. ID рабочего процесса находится в URL:

```
https://www.sim.ai/workspace/{workspaceId}/w/{workflowId}
```

Вы также можете использовать эндпоинт [List Workflows](/api-reference/workflows/listWorkflows) для получения всех ID рабочих процессов в рабочей области.
</Step>

<Step>
### Разверните рабочий процесс

Рабочий процесс должен быть развёрнут, прежде чем его можно будет выполнить через API. Нажмите кнопку **Deploy** на панели инструментов редактора или используйте панель управления для управления развёртываниями.
</Step>

<Step>
### Выполните первый запрос

<Tabs items={['curl', 'TypeScript', 'Python']}>

      -d '{"inputs": {}}'

## Синхронное и асинхронное выполнение

По умолчанию выполнение рабочих процессов является **синхронным** — API блокируется до завершения рабочего процесса и возвращает результат напрямую.

Для длительных рабочих процессов используйте **асинхронное выполнение**, передав `async: true`:

```bash
curl -X POST https://www.sim.ai/api/workflows/{workflowId}/execute \
  -H "Content-Type: application/json" \
  -H "X-API-Key: YOUR_API_KEY" \
  -d '{"inputs": {}, "async": true}'
```

Запрос возвращается немедленно с `jobId` и `statusUrl`:

```json
{
  "success": true,
  "jobId": "job_abc123",
  "statusUrl": "https://www.sim.ai/api/jobs/job_abc123",
  "message": "Workflow execution started",
  "async": true
}
```

Опрашивайте эндпоинт [Get Job Status](/api-reference/workflows/getJobStatus), пока статус не станет `completed` или `failed`:

```bash
curl https://www.sim.ai/api/jobs/{jobId} \
  -H "X-API-Key: YOUR_API_KEY"
```

<Callout type="info">
  Переходы статуса задания: `queued` → `processing` → `completed` или `failed`. Поле `output` присутствует только при статусе `completed`.
</Callout>

## Формат ответа

Успешные ответы включают объект `output` с результатами вашего рабочего процесса и объект `limits` с текущим состоянием лимита запросов и использования:

```json
{
  "success": true,
  "output": {
    "result": "Hello, world!"
  },
  "limits": {
    "workflowExecutionRateLimit": {
      "sync": {
        "requestsPerMinute": 60,
        "maxBurst": 10,
        "remaining": 59,
        "resetAt": "2025-01-01T00:01:00Z"
      },
      "async": {
        "requestsPerMinute": 30,
        "maxBurst": 5,
        "remaining": 30,
        "resetAt": "2025-01-01T00:01:00Z"
      }
    },
    "usage": {
      "currentPeriodCost": 1.25,
      "limit": 50.00,
      "plan": "pro",
      "isExceeded": false
    }
  }
}
```

## Обработка ошибок

API использует стандартные коды состояния HTTP. Ответы с ошибками включают читаемое сообщение `error`:

```json
{
  "error": "Workflow not found"
}
```

| Статус | Значение | Что делать |
| --- | --- | --- |
| `400` | Неверные параметры запроса | Проверьте массив `details` на наличие конкретных ошибок полей |
| `401` | Отсутствует или неверный API-ключ | Проверьте заголовок `X-API-Key` |
| `403` | Доступ запрещён | Убедитесь, что у вас есть разрешение на этот ресурс |
| `404` | Ресурс не найден | Проверьте, что ID существует и принадлежит вашей рабочей области |
| `429` | Превышен лимит запросов | Подождите в течение времени, указанного в заголовке `Retry-After` |

<Callout type="info">
  Используйте эндпоинт [Get Usage Limits](/api-reference/usage/getUsageLimits), чтобы в любое время проверить текущее состояние лимита запросов и использование биллинга.
</Callout>

## Лимиты запросов

Лимиты запросов зависят от вашего тарифного плана и применяются отдельно к синхронным и асинхронным выполнениям. Каждый ответ выполнения включает объект `limits`, показывающий текущее состояние вашего лимита запросов.

При превышении лимита API возвращает ответ `429` с заголовком `Retry-After`, указывающим, сколько секунд подождать перед повторной попыткой.

## Пагинация

Эндпоинты списков (рабочие процессы, журналы, аудит) используют **курсорную пагинацию**:

```bash
# Первая страница
curl "https://www.sim.ai/api/v1/logs?limit=20" \
  -H "X-API-Key: YOUR_API_KEY"

# Следующая страница — используйте nextCursor из предыдущего ответа
curl "https://www.sim.ai/api/v1/logs?limit=20&cursor=abc123" \
  -H "X-API-Key: YOUR_API_KEY"
```

Ответ включает поле `nextCursor`. Если `nextCursor` отсутствует или равен `null`, вы достигли последней страницы.
