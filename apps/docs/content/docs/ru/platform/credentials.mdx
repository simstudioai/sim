Секреты — это пары ключ-значение, в которых хранятся конфиденциальные данные, такие как ключи API, токены и пароли. Вместо того чтобы жёстко кодировать значения в рабочих процессах, вы сохраняете их как секреты и ссылаетесь на них по имени во время выполнения.

## Управление секретами

Для управления секретами откройте **Настройки** рабочего пространства и перейдите на вкладку **Секреты**.

<Image
  src="/static/secrets/secrets-list.png"
  alt="Вкладка Секреты, показывающая разделы Рабочее пространство и Личные со строками ключ-значение"
  width={700}
  height={500}
/>

Секреты организованы в два раздела:

- **Рабочее пространство** — общие для всех участников вашего рабочего пространства
- **Личные** — приватные, только для вас

Внешние участники рабочего пространства считаются участниками рабочего пространства для секретов уровня рабочего пространства. Они могут использовать секреты рабочего пространства в соответствии со своим уровнем разрешений, даже если они не являются участниками вашей организации.

### Добавление секрета

Введите имя ключа (например, `OPENAI_API_KEY`) в колонку **Key** и его значение в колонку **Value** в последней пустой строке. Новая пустая строка появляется автоматически по мере ввода. Существующие значения по умолчанию скрыты.

Когда закончите, нажмите **Save**, чтобы сохранить все изменения.

<Callout type="info">
Ключи должны использовать только буквы, цифры и подчёркивания — без пробелов и специальных символов.
</Callout>

### Массовый импорт

Вы можете заполнить несколько секретов одновременно, вставив содержимое в формате `.env` в любое поле ключа или значения. Парсер поддерживает стандартные пары `KEY=VALUE`, `export KEY=VALUE`, значения в кавычках и встроенные комментарии.

### Редактирование и удаление

Нажмите непосредственно на любую ячейку ключа или значения для редактирования. Чтобы удалить секрет, нажмите значок корзины в его строке и сохраните.

## Использование секретов в рабочих процессах

Чтобы сослаться на секрет в любом поле ввода, введите `{{`, чтобы открыть выпадающий список переменных. Доступные секреты перечислены с группировкой по области действия (рабочее пространство, затем личные).

Выберите нужный секрет. Ссылка отображается выделенной синим цветом и разрешается в фактическое значение во время выполнения.

<Image

<Callout type="warn">
Значения секретов никогда не раскрываются в редакторе рабочих процессов или журналах выполнения — они разрешаются только во время выполнения.
</Callout>

## Детали секрета

Нажмите **Details** на любой строке секрета, чтобы открыть его детальный вид.

<Image
  src="/static/secrets/secret-details.png"
  alt="Детальный вид секрета, показывающий разделы Отображаемое имя, Описание и Участники"
  width={700}
  height={400}
/>

Отсюда вы можете:

- Редактировать **Отображаемое имя** и **Описание**
- Управлять **Участниками** — приглашать коллег по email и назначать им роль **Admin** или **Member**

Нажмите **Save** для применения изменений или **Back** для возврата к списку.

## Рабочее пространство vs. Личные

| | Рабочее пространство | Личные |
|---|---|---|
| **Видимость** | Все участники рабочего пространства, включая внешних участников | Только вы |
| **Использование в рабочих процессах** | Любой участник может использовать | Только вы можете использовать |
| **Лучше всего для** | Рабочих процессов в продакшене, общих сервисов | Тестирования, личных ключей API |
| **Кто может редактировать** | Администраторы рабочего пространства | Только вы |

<Callout type="info">
Когда секрет рабочего пространства и личный секрет имеют одинаковое имя ключа, **секрет рабочего пространства имеет приоритет**.
</Callout>

### Порядок разрешения

При выполнении рабочего процесса секреты разрешаются в следующем порядке:

1. **Секреты рабочего пространства** проверяются первыми
2. **Личные секреты** используются как резервный вариант — от пользователя, запустившего выполнение (ручной запуск), или от владельца рабочего процесса (автоматические запуски через API, вебхук или расписание)

## Рекомендации

- **Используйте секреты рабочего пространства для продакшена**, чтобы рабочие процессы работали независимо от того, кто их запускает
- **Используйте личные секреты для разработки**, чтобы хранить тестовые ключи отдельно
- **Называйте ключи описательно** — `STRIPE_SECRET_KEY` вместо `KEY1`
- **Никогда не жёстко кодируйте секреты** в полях ввода рабочих процессов — всегда используйте ссылки `{{KEY}}`
