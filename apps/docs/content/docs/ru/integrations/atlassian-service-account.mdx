Сервисные аккаунты Atlassian позволяют вашим рабочим процессам аутентифицироваться в Jira и Confluence как нечеловеческий бот-пользователь — независимо от учётной записи любого отдельного сотрудника. Каждый сервисный аккаунт имеет свой собственный адрес электронной почты, свои разрешения и свои API-токены, управляемые централизованно в admin.atlassian.com.

Это рекомендуемый способ использования Jira и Confluence в производственных рабочих процессах: ни у кого не истекает срок действия OAuth-согласия, разрешения бота поддаются аудиту, а доступ может быть отозван без затрагивания чьей-либо личной учётной записи.

## Предварительные требования

Для создания сервисного аккаунта требуется администратор организации Atlassian. Сервисные аккаунты — это функция уровня организации Atlassian, их невозможно создать из обычной учётной записи пользователя.

## Настройка сервисного аккаунта

### 1. Создание сервисного аккаунта

<Steps>
  <Step>
    Откройте [admin.atlassian.com](https://admin.atlassian.com/) и перейдите в **Directory** → **Service accounts**

    {/* TODO(screenshot): admin.atlassian.com directory page with the "Service accounts" tab highlighted */}
  </Step>
  <Step>
    Нажмите **Create service account**, присвойте ему имя (например, `sim-jira-bot`) и завершите создание
  </Step>
  <Step>
    Предоставьте сервисному аккаунту доступ к необходимым сайтам и продуктам Atlassian. Откройте сервисный аккаунт, перейдите в **Product access** и добавьте Jira и/или Confluence на соответствующем сайте

    {/* TODO(screenshot): service account "Product access" tab showing Jira granted on a site */}
  </Step>
</Steps>

<Callout type="info">
Сервисный аккаунт наследует разрешения от ролей проекта/пространства, которые вы ему предоставляете — точно так же, как обычный пользователь. Если рабочему процессу требуется запись в конкретный проект Jira, предоставьте сервисному аккаунту доступ на запись к этому проекту в настройках проекта Jira.
</Callout>

### 2. Создание ограниченного API-токена

<Steps>
  <Step>
    На странице сервисного аккаунта в admin.atlassian.com откройте вкладку **API tokens** и нажмите **Create API token**

    {/* TODO(screenshot): service account API tokens tab with "Create API token" button */}
  </Step>
  <Step>
    Выберите **API token** в качестве типа аутентификации (не OAuth 2.0 — Sim использует поток API-токена)

    <div className="flex justify-center">
      <Image
        src="/static/credentials/atlassian/admin-auth-type-picker.png"
        alt="Atlassian admin — Choose authentication type with API token selected"
        width={700}
        height={500}
        className="my-4"
      />
    </div>
  </Step>
  <Step>
    Выберите области действия (scopes), необходимые токену. Минимальный набор, ожидаемый блоками Jira и Confluence в Sim:

    **Jira (classic):**
    ```
    read:jira-user
    read:jira-work
    write:jira-work
    ```

    **Jira Service Management (classic):**
    ```
    read:servicedesk-request
    write:servicedesk-request
    manage:servicedesk-customer
    ```

    **Confluence (granular):**
    ```
    read:confluence-content.all
    read:confluence-space.summary
    write:confluence-content
    read:page:confluence
    write:page:confluence
    ```

    Добавляйте дополнительные области действия только если вам нужны соответствующие операции (удаление, управление вебхуками и т.д.). Полный список областей действия, которые могут использовать блоки Sim, описан в [справочной документации Atlassian для разработчиков](https://developer.atlassian.com/cloud/jira/platform/scopes-for-oauth-2-3LO-and-forge-apps/).

    <Callout type="warn">
      Предпочитайте классические (classic) области действия, указанные выше, вместо детализированных (granular) эквивалентов. Atlassian применяет список детализированных областей действия конечной точки по принципу «всё или ничего», поэтому токен, созданный из частичного набора детализированных областей, завершится ошибкой `Unauthorized; scope does not match`, даже если каждая отдельная область была предоставлена. Классические области действия покрывают конечные точки своего продукта самостоятельно. Если ваша организация разрешает только детализированные области действия, включите все области, перечисленные для каждой конечной точки в справочнике Atlassian — операции Jira Service Management также требуют `read:user:jira`.
    </Callout>

    <div className="flex justify-center">
      <Image
        src="/static/credentials/atlassian/admin-scope-picker.png"
        alt="Atlassian token scope picker filtered to App: Jira and Scope type: Classic"
        width={1000}
        height={600}
        className="my-4"
      />
    </div>

    <Callout type="info">
      Используйте фильтры **App** и **Scope type**, чтобы сузить список до нужных вам областей действия. Фильтруйте по `App: Jira` (или `Confluence`) и `Scope type: Classic`, чтобы найти три основные области действия Jira; переключитесь на **Granular**, если ваша организация не предоставляет Classic.
    </Callout>
  </Step>
  <Step>
    Скопируйте токен при его отображении. Atlassian показывает его только один раз — если вы закроете диалоговое окно, вам придётся создать новый токен.
  </Step>
</Steps>

<Callout type="warn">
API-токен является носителем учётных данных (bearer credentials) для сервисного аккаунта. Обращайтесь с ним как с паролем — не добавляйте его в систему контроля версий и не публикуйте公开но. Sim шифрует токен при хранении.
</Callout>

### 3. Найдите домен вашего сайта

Домен вашего сайта Atlassian — это URL, который вы используете для доступа к Jira или Confluence в браузере — например, `your-team.atlassian.net`. Откройте Jira или Confluence, посмотрите на адресную строку и скопируйте часть до первого `/`.

## Добавление сервисного аккаунта в Sim

<Steps>
  <Step>
    Откройте **Настройки** рабочего пространства и перейдите на вкладку **Integrations**
  </Step>
  <Step>
    Найдите "Atlassian Service Account" и нажмите на него

    {/* TODO(screenshot): Integrations page with "Atlassian Service Account" in the service list */}
  </Step>
  <Step>
    Вставьте API-токен, введите домен сайта (например, `your-team.atlassian.net`) и при необходимости задайте отображаемое имя и описание

    <div className="flex justify-center">
      <Image
        src="/static/credentials/atlassian/sim-add-modal.png"
        alt="Add Atlassian Service Account dialog with API token and site domain filled in"
        width={420}
        height={560}
        className="my-6"
      />
    </div>
  </Step>
  <Step>
    Нажмите **Add Service Account**. Sim проверяет токен, вызывая конечную точку Atlassian `/myself` через шлюз — в случае неудачи вы увидите конкретную ошибку с объяснением причины.
  </Step>
</Steps>

Токен, домен и обнаруженный cloudId шифруются перед сохранением.

## Использование сервисного аккаунта в рабочих процессах

Добавьте блок Jira или Confluence в ваш рабочий процесс. В выпадающем списке учётных данных ваш сервисный аккаунт Atlassian отображается наряду с любыми учётными данными OAuth. Выберите его и настройте блок как обычно.

<div className="flex justify-center">
  <Image
    src="/static/credentials/atlassian/sim-jira-block-credential.png"
    alt="Jira block in a workflow with the Atlassian service account selected as the credential"
    width={1000}
    height={500}
    className="my-4"
  />
</div>

Блок вызывает шлюз API Atlassian (`api.atlassian.com/ex/jira/{cloudId}/...`), используя токен сервисного аккаунта. Шаг имперсонации отсутствует — сервисный аккаунт действует от своего имени с теми разрешениями, которые вы предоставили ему в admin.atlassian.com.
