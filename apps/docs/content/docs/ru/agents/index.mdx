**Агент** — это [рабочий процесс](/workflows), который самостоятельно рассуждает и действует. Он читает входные данные, решает, что делать, и выполняет это, вызывая инструменты и используя ваши данные по ходу работы. Вы создаёте пользовательского агента в Sim, составляя рабочий процесс, в котором рассуждение проходит через один или несколько [блоков Agent](/workflows/blocks/agent).

**Блок Agent** — это шаг рассуждения внутри этого воркфлоу: модель читает доступные ей значения, принимает решение и возвращает результат, который последующие блоки читают по ссылке. Простой агент — это один блок Agent; более крупный соединяет несколько вместе с другими блоками. Блок Agent — это место, где модель думает; остальная часть воркфлоу — это то, над чем и через что она действует.

В качестве примера на протяжении всего текста используется агент, который оценивает входящие лиды продаж.

## Блок Agent

Вы настраиваете шаг рассуждения, давая блоку Agent **модель** и **промпт**. Модель — это LLM, которая его питает; вы выбираете одну из доступных провайдеров, и по умолчанию используется `claude-sonnet-4-6`. Промпт — это системное сообщение, которое определяет, кем является агент и как он должен себя вести, плюс пользовательское сообщение, которое несёт входные данные, обычно ссылку вроде `<start.input>`.

Когда он выполняется, блок Agent рассуждает, вызывает любые необходимые инструменты и сохраняет свой результат под собственным именем. По умолчанию этот результат — свободный текст в `content`, читаемый последующим блоком как `<agent.content>`, вместе с деталями запуска, такими как использованная модель, количество токенов, вызовы инструментов и стоимость. Все настройки и выходные поля описаны в [справочнике блока Agent](/workflows/blocks/agent).

## Что вы даёте агенту

Сам по себе блок Agent может только рассуждать и писать текст. Вы расширяете его, чтобы он мог действовать, следовать вашим правилам, использовать ваши данные, запоминать и возвращать результаты, на которые могут положиться другие блоки. Каждая возможность соответствует тому, что вы хотели бы, чтобы агент делал.

### Совершить действие: инструменты

Чтобы позволить агенту делать что-то в мире, дайте ему **инструменты**. Инструмент — это действие, которое агент может вызвать, например, отправка письма, поиск в интернете, обновление записи в CRM или запуск другого воркфлоу. Вы прикрепляете инструменты к блоку Agent, и агент решает, какой из них вызвать для текущей задачи. В оценщике лидов у агента есть инструмент поиска для сбора контекста по скудному профилю и инструмент отправки письма для связи с перспективным лидом.

Инструменты поступают из нескольких источников:

- **[Интеграции](/integrations)** — это каталог внешних сервисов: Gmail, Slack, Airtable, Linear и сотни других.
- **[Пользовательские инструменты](/agents/custom-tools)** — это инструменты, которые вы определяете один раз со схемой и фрагментом кода, а затем переиспользуете.
- **[MCP-инструменты](/agents/mcp)** поступают от внешнего провайдера, которого вы подключаете через Model Context Protocol.
- **[Воркфлоу как инструмент](/workflows)** делает другой воркфлоу вызываемым, так что агент выполняет целую процедуру как один шаг.

<Callout type="info">
Одна и та же интеграция может работать двумя способами. Как [блок](/workflows#blocks) она выполняется, потому что путь дошёл до неё. Как инструмент агента она выполняется, потому что агент выбрал её. Элементы управления использованием инструментов (принудительный вызов или отключение) описаны в [справочнике блока Agent](/workflows/blocks/agent).
</Callout>

### Следовать процедуре: навыки

Чтобы дать агенту инструкции, которым он может следовать, напишите [навык](/agents/skills). Навык — это переиспользуемое руководство с коротким именем и описанием, которые агент всегда видит, а также более длинным телом, которое он загружает только тогда, когда навык применим. В оценщике лидов навык `lead-scoring-rubric` описывает диапазоны оценок и критерии дисквалификации, и агент читает полную рубрику только тогда, когда лид неоднозначен. Инструмент — это действие, которое агент совершает; навык — это руководство, которое он читает.

### Использовать ваши документы: знания

Чтобы позволить агенту отвечать на основе вашего собственного контента, подключите [базу знаний](/knowledgebase). Агент ищет в ней и обосновывает свои ответы тем, что находит, вместо того чтобы полагаться только на общее обучение модели. Дайте оценщику лидов базу знаний о прошлых сделках, и он сможет сравнить новый лид с теми, которые вы уже закрыли.

### Запоминать между запусками: память

Чтобы позволить агенту переиспользовать информацию от одного запуска к другому, дайте ему [память](/integrations/memory), которая сохраняет и извлекает значения, привязанные к разговору. Без неё каждый запуск начинается с чистого листа; с ней агент в чате переносит сказанное ранее в последующие сообщения.

### Возвращать пригодный результат: структурированный вывод

Чтобы сделать результат агента чем-то, с чем могут работать последующие блоки, дайте ему **структурированный вывод**: типизированный объект, который вы определяете, вместо свободного текста. В оценщике лидов агент возвращает `{ score, tier, reason }`, и последующий блок [Condition](/workflows/blocks/condition) читает `<agent.tier>` для ветвления. См. [как блоки передают данные](/workflows/data-flow) для чтения полей.

## Выбор того, что использовать

Начните с одного блока Agent и промпта, затем добавляйте только то, что требуется задаче: инструмент, когда агент должен действовать, навык, когда ему нужно письменное руководство, базу знаний, когда он должен отвечать на основе ваших документов, память, когда он должен запоминать, и структурированный вывод, когда последующий блок должен прочитать его результат.
