При создании агента несколько функций пересекаются: детерминированный блок и инструмент агента могут выполнять одну и ту же интеграцию, а также пользовательский инструмент, инструмент MCP и рабочий процесс в качестве инструмента могут предоставлять агенту одинаковое действие. На этой странице изложены различия, чтобы вы могли выбрать наиболее подходящий вариант. Они различаются по трем параметрам: является ли действие **детерминированным** (всегда выполняется) или **определяется моделью** (агент выбирает), находится ли оно в одном рабочем процессе или может быть использовано в вашем рабочем пространстве, и исходит ли оно от Sim или **внешнего** поставщика.

Приведённый пример – это рабочий процесс, который оценивает входящие лиды продаж. Он считывает новый лид, обогащает его информацией, определяет оценку, записывает результат и уведомляет команду. Каждый из вариантов ниже является частью этого процесса:

## Определённый блок
**Блок** – это отдельный шаг, который выполняется в определенной точке пути, без необходимости принятия решения моделью. Он всегда выполняется, когда рабочий процесс достигает этого блока. Используйте его, когда действие должно выполняться каждый раз: вызов API, преобразование данных, переход к другой ветке.

В списке лучших игроков, в[Таблица Google](/integrations)Таблица Google/интеграцииблок всегда добавляет оцененного лида в таблицу отслеживания, и[Функция](/workflows/blocks/function)Функция/рабочие процессы/блоки/функцияБлок всегда преобразует ответ обогащения в поля, которые ожидает следующий этап. Здесь нет никаких оценочных суждений. Работа гарантирована.

<WorkflowPreview workflow={LEAD_SCORER_WORKFLOW} height={170} highlightBlock="log" />

Большинство шагов в рабочем процессе представляют собой отдельные блоки. Используйте только те типы ниже, когда вам нужно, чтобы модель принимала решение, или когда вы хотите повторно использовать что-то.

## Инструмент для агентов
**Инструмент для агента** – это действие, которое вы передаете агенту.[Представитель](/workflows/blocks/agent)Представитель/рабочие процессы/блоки/агентблок. Агент читает задачу и решает, нужно ли ее выполнять и когда. То же относится к каталогу[интеграции](/integrations)интеграции/интеграциикоторые существуют в виде отдельных блоков, также могут быть присоединены к агенту в качестве инструментов.

В разделе "Лиды", агент имеет инструменты для поиска и отправки электронных писем. Для лида с неполной информацией агент использует инструмент поиска для получения контекстной информации; для сильного лида – инструмент отправки электронных писем. В случае слабого, очевидного лида может срабатывать только один из инструментов. Выбор делается в каждом конкретном случае.

<WorkflowPreview workflow={LEAD_SCORER_WORKFLOW} height={170} highlightBlock="score" />

Каждый инструмент имеет настройку `usageControl`. **Автоматический режим** позволяет модели самостоятельно решать (это значение по умолчанию). **Принудительный режим** заставляет агента использовать этот инструмент при каждом выполнении, для действий, которые никогда не должны быть пропущены, например, всегда записывать принятое решение. **Отсутствие** удаляет этот инструмент из данного агента.

<Callout type="info">
A block and an agent tool can be the same underlying integration. The difference is who decides. A block runs because the path reached it. An agent tool runs because the agent chose it.
</Callout>

## Инструмент по индивидуальному заказу
**Настраиваемый инструмент** – это инструмент, который вы определяете один раз с помощью схемы объекта и фрагмента кода на JavaScript или Python, а затем используете в своем рабочем пространстве. Ему не требуется внешний аккаунт. Используйте его, когда вам нужно реализовать логику, которую иначе должны бы дублировать несколько агентов или рабочих процессов.

В разделе "Лидер", специальный инструмент `normalizeCompanyDomain` очищает исходный веб-сайт и преобразует его в стандартный домен. Этот же инструмент используется для определения лидера, рабочего процесса дублирования и агента отчетности. Определите его в рабочей области, а затем выберите из списка инструментов любого блока агента.

## Сервер MCP
**MCP** (Протокол контекста модели) – это стандарт для подключения стороннего поставщика инструментов. Подключите его[Сервер MCP](/agents/mcp)Сервер MCP/агенты/mcpа его инструменты отображаются в списке инструментов агента как единый набор. Используйте его для импорта полного набора инструментов, который Sim не предоставляет по умолчанию, вместо ручной настройки каждого действия.

В системе для ведущего игрока ваш поставщик CRM предоставляет сервер MCP. После того, как вы настроите его один раз, агент сможет читать учетные записи и обновлять данные с помощью собственных инструментов поставщика. Разница между этим и инструментом, разработанным на заказ, заключается в том, кто его поддерживает: инструмент, разработанный на заказ, – это код, который вы написали, а сервер MCP – это набор инструментов, которые поддерживает другой человек.

## Рабочий процесс как инструмент
**Рабочий процесс как инструмент** – это весь рабочий процесс, который передается агенту в виде одного вызываемого инструмента. Вы выбираете рабочий процесс в…[Представитель](/workflows/blocks/agent)Представитель/рабочие процессы/блоки/агентсписок инструментов для блока; агент решает, когда его использовать, и предоставляет необходимые данные, которые поступают к ребенку.[Начните](/workflows/triggers/start)Начните/рабочие процессы/триггеры/старттриггер, и результат работы ребенка возвращается в качестве вывода инструмента. Используйте его, когда необходимо, чтобы многоступенчатая процедура была доступна агенту, а не выполнялась автоматически.

В случае с лидирующим потенциальным клиентом, агент использует `Deep Enrich` как инструмент – свою пятиступенчатую процедуру. Для слабого лида агент использует ее для заполнения профиля перед оценкой; для полного лида она не используется. Агент рассматривает всю процедуру так же, как и отдельное действие.

<Callout type="info">
A workflow can also run as a fixed step: the [Workflow](/workflows/blocks/workflow) block calls a child workflow because the path reached it. Same child workflow, same Start trigger — the difference, as with blocks and agent tools, is who decides. The Enrich step in the diagram is that deterministic case.
</Callout>

## Навык
**Навык** – это переиспользуемые инструкции, письменный гайд, который агент может использовать. Каждый навык имеет короткое название и описание, которые всегда видны агенту, а также более подробное описание, которое агент загружает только тогда, когда решает, что этот навык применим. Используйте его для описания того, как что-то нужно делать, отдельно от инструментов, которые это делают.

В поле «Лучший игрок», указанная `lead-scoring-rubric` указывает на команды и критерии отбора. Агент видит название и описание навыка при каждом выполнении, а также загружает полный список критериев и применяет его, когда оценка неоднозначна. Различие простое: инструмент – это действие, которое совершает агент, а навык – это руководство, которое он читает. Управляйте навыками в вашем[рабочее пространство](/agents/skills)рабочее пространство/навыки агентов.

## Собираем все вместе
Главный снайпер использует шесть типов одновременно: блоки для шагов, которые всегда должны выполняться, инструменты для работы агента для звонков, которые модель должна учитывать, пользовательский инструмент для общей логики, сервер MCP для CRM, рабочий процесс глубокого обогащения, который агент использует, когда ему это необходимо, и навык для правил оценки. В общем, начните с блока, затем перейдите к инструменту для работы агента, когда модель должна принять решение. Используйте остальные только тогда, когда вам это необходимо: внешний набор инструментов, подпроцесс или письменные инструкции.
