**Блок Condition** ветвит воркфлоу по булевым выражениям. Он проверяет каждую ветку по порядку и выполняет путь первой истинной, без вызова модели. Если вы хотите, чтобы модель выбирала маршрут, используйте [Router](/workflows/blocks/router).

<BlockPreview type="condition" />

## Конфигурация

### Conditions

Список веток: **if**, любое количество строк **else if** и финальный **else**. Каждая ветка содержит JavaScript-выражение, которое вычисляется в true или false, и подключается к своему пути. Sim проверяет их сверху вниз и выбирает первую истинную ветку; ветка **else** выполняется, когда ни одна не совпала.

Ссылайтесь на предыдущий output внутри выражения с помощью [тега соединения](/workflows/connections):

```javascript
<agent.score> > 75
<ticket.subject>.toLowerCase().includes('urgent')
<user.email>.endsWith('@company.com') && <user.plan> === 'pro'
```

<Callout type="info">
Если выражение выбрасывает исключение, например, потому что читает отсутствующее поле, блок завершается с ошибкой, и запуск идёт по [пути ошибки](/workflows/connections), если он подключён. Защитите отсутствующие значения с помощью опциональной цепочки (`?.`) или проверки на null.
</Callout>

## Output'ы

| Output | Что это |
| --- | --- |
| `<condition.conditionResult>` | Булев результат совпавшей ветки |
| `<condition.selectedOption>` | Идентификатор совпавшей ветки |
| `<condition.selectedPath>` | Целевой блок, к которому направился воркфлоу |

## Примеры

### Маршрутизация тикета поддержки по приоритету

<WorkflowPreview workflow={CONDITION_ROUTE_WORKFLOW} />

Condition проверяет `<start.priority>` и выполняет один из downstream-путей. Другой не выполняется, поэтому ссылка на блок на невыполненном пути возвращает пустое значение.

### Модерация контента по оценке

<WorkflowPreview workflow={CONDITION_MODERATE_WORKFLOW} />

Agent оценивает контент, а Condition публикует или блокирует его на основе `<moderate.toxicity>`.

### Ветвление онбординга по уровню аккаунта

<WorkflowPreview workflow={CONDITION_ONBOARD_WORKFLOW} />

Function читает аккаунт, а Condition направляет enterprise-аккаунты через управляемую настройку, а всех остальных — через быстрый старт.

## Рекомендации

- **Упорядочивайте ветки от частного к общему.** Первая истинная ветка побеждает, поэтому размещайте узкие проверки перед общими.
- **Всегда давайте путь для несовпавших запусков.** Подключайте ветку **else** или делайте последнее условие `true`, чтобы ничего не уходило в пустоту молча.
- **Держите выражения простыми.** Короткие булевы проверки легко читать и отлаживать. Тяжёлую логику выполняйте в блоке [Function](/workflows/blocks/function) и ветвитесь по его результату.
- **Защищайте отсутствующие значения.** Используйте опциональную цепочку (`?.`) или проверку на null, чтобы отсутствующее поле не вызывало исключение.
