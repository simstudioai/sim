**Блок Evaluator** использует модель для оценки контента по заданным вами метрикам, возвращая число для каждой метрики. Используйте его для контроля качества, сравнения вариантов и проверки AI-вывода.

## Конфигурация

### Evaluation Metrics

Метрики для оценки. Каждая метрика имеет **имя**, **описание** того, что она измеряет, и числовой **диапазон**:

```
Accuracy (1-5): Насколько фактически точен контент?
Clarity  (1-5): Насколько он ясен и понятен?
Relevance(1-5): Насколько он релевантен исходному запросу?
```

Модель оценивает контент по каждой метрике и возвращает числа. Метрики без имени или диапазона пропускаются.

### Content

Контент для оценки. Обычно это предыдущий output, например `<agent.content>`. Структурированные данные форматируются в текст перед оценкой; оценка текстовая, поэтому она не может напрямую оценивать изображения или аудио.

### Model

Модель, выполняющая оценку, по умолчанию `claude-sonnet-4-6`. Более сильные модели рассуждения дают более стабильные оценки. Введите или выберите любую поддерживаемую модель. **Temperature** и **System Prompt** доступны в расширенных настройках, а на хостинге Sim API-ключ предоставляется автоматически.

## Output'ы

Evaluator возвращает число для каждой метрики, читаемое по имени метрики в нижнем регистре:

| Output | Что это |
| --- | --- |
| `<evaluator.accuracy>` | Оценка по метрике (один output на каждую заданную метрику) |
| `<evaluator.content>` | Сводка оценки |
| `<evaluator.model>` | Модель, выполнившая оценку |
| `<evaluator.tokens>` | Использование токенов |
| `<evaluator.cost>` | Оценка стоимости вызова |

Блок принудительно использует JSON Schema, построенную на основе ваших метрик, поэтому модель возвращает только оценки метрик как числа, без лишнего текста.

## Примеры

### Проверка по порогу качества

<WorkflowPreview workflow={EVALUATOR_GATE_WORKFLOW} />

Evaluator оценивает черновик, а [Condition](/workflows/blocks/condition) проверяет `<evaluator.accuracy>` — публикуя сильный черновик или отправляя слабый обратно на доработку.

Та же схема подходит для других задач качества: оцените несколько [параллельных](/workflows/blocks/parallel) вариантов и выберите лучший или оценивайте каждый ответ поддержки и помечайте слабые для проверки.

## Рекомендации

- **Пишите конкретные описания метрик.** Чёткое определение того, что измеряет каждая метрика, даёт более стабильные оценки.
- **Выбирайте разумный диапазон.** Достаточная гранулярность для принятия решений (1–5 или 0–10), без излишней детализации.
- **Оценивайте output Agent и возвращайте на доработку.** Совместите Evaluator с Condition для проверки порога и направления слабого вывода на ещё один проход.
- **Сохраняйте метрики единообразными.** При сравнении вариантов используйте одни и те же метрики для каждой оценки.
