**Блок Human in the Loop** приостанавливает запуск и ждёт человека перед продолжением. Используйте его для шлюзов подтверждения, сбора обратной связи или получения ввода в точке принятия решения. Запуск остаётся приостановленным — без тайм-аута — пока кто-то не ответит через портал подтверждения, API или webhook.

## Конфигурация

### Display Data

Что видит утверждающий — контекст, показанный в портале для помощи в принятии решения. Соберите его по полям или как JSON, ссылаясь на предыдущие output'ы через `<имяБлока.output>`.

```json
{
  "customerName": "<agent1.content.name>",
  "proposedAction": "<router1.selectedPath>",
  "confidenceScore": "<evaluator1.score>",
  "generatedEmail": "<agent2.content>"
}
```

### Notification

Как утверждающие оповещаются о том, что решение ожидает. Включите URL подтверждения (`<blockId.url>`) в сообщение, чтобы они могли открыть портал. Доступные каналы:

- **Slack** — сообщение в канал или личное сообщение
- **Gmail** — письмо со ссылкой подтверждения
- **Microsoft Teams** — уведомление в канал
- **SMS** — текстовое оповещение через Twilio
- **Webhook** — запрос в вашу собственную систему уведомлений

### Resume Form

Поля, которые утверждающий заполняет при ответе. Каждое становится доступным для последующих блоков после возобновления запуска.

```json
{
  "approved": {
    "type": "boolean",
    "description": "Одобрить или отклонить этот запрос"
  },
  "comments": {
    "type": "string",
    "description": "Дополнительная обратная связь или объяснение"
  }
}
```

Доступ к данным возобновления в последующих блоках через `<blockId.fieldName>`.

## Методы подтверждения

<Tabs items={['Approval Portal', 'API', 'Webhook']}>
  <Tab>
    ### Портал подтверждения
    
    Каждый блок генерирует уникальный URL портала (`<blockId.url>`) с визуальным интерфейсом, показывающим все приостановленные выходные данные и поля формы для ввода при возобновлении. Адаптивный для мобильных устройств и безопасный.
    
    Отправьте этот URL в уведомлениях, чтобы утверждающие могли просмотреть и ответить.
  </Tab>
  <Tab>
    ### REST API
    
    Программное возобновление воркфлоу через эндпоинт resume. `contextId` доступен из output'а `resumeEndpoint` блока или из объекта `_resume` в ответе приостановленного выполнения.
    
    ```bash
    POST /api/resume/{workflowId}/{executionId}/{contextId}
    Content-Type: application/json
    X-API-Key: your-api-key
    
    {
      "input": {
        "approved": true,
        "comments": "Выглядит хорошо, можно продолжать"
      }
    }
    ```
    
    Эндпоинт resume автоматически учитывает режим выполнения, использованный в исходном вызове execute:
    
    - **Режим Sync** (по умолчанию) — Ответ ждёт завершения оставшегося воркфлоу и возвращает полный результат:
    
    ```json
    {
      "success": true,
      "status": "completed",
      "executionId": "<resumeExecutionId>",
      "output": { ... },
      "metadata": { "duration": 1234, "startTime": "...", "endTime": "..." }
    }
    ```
    
    Если возобновлённый воркфлоу достигает другого блока HITL, ответ возвращает `"status": "paused"` с новыми URL `_resume` в выводе.
    
    - **Режим Stream** (`stream: true` в исходном вызове execute) — Ответ resume передаёт события SSE с фрагментами `selectedOutputs`, как и исходное выполнение.
    
    - **Режим Async** (`X-Execution-Mode: async` в исходном вызове execute) — Resume отправляет выполнение фоновому обработчику и немедленно возвращает `202` с `jobId` и `statusUrl` для опроса:
    
    ```json
    {
      "success": true,
      "async": true,
      "jobId": "<jobId>",
      "executionId": "<resumeExecutionId>",
      "message": "Resume execution queued",
      "statusUrl": "/api/jobs/<jobId>"
    }
    ```
    
    #### Опрос статуса выполнения
    
    Опрашивайте `statusUrl` из асинхронного ответа для проверки завершения resume:
    
    ```bash
    GET /api/jobs/{jobId}
    X-API-Key: your-api-key
    ```
    
    Возвращает статус задачи и, по завершении, полный вывод воркфлоу.
    
    Для проверки точек паузы приостановленного выполнения и ссылок resume:
    
    ```bash
    GET /api/resume/{workflowId}/{executionId}
    X-API-Key: your-api-key
    ```
    
    Возвращает детали приостановленного выполнения со всеми точками паузы, их статусами и ссылками resume. Возвращает `404`, когда выполнение завершено и больше не приостановлено.
  </Tab>
  <Tab>
    ### Webhook
    
    Добавьте инструмент webhook в раздел Notification для отправки запросов подтверждения во внешние системы. Интегрируйтесь с тикет-системами, такими как Jira или ServiceNow.
  </Tab>
</Tabs>

## Поведение API Execute

При запуске воркфлоу через API execute (`POST /api/workflows/{id}/execute`) блоки HITL вызывают приостановку выполнения и возвращают данные `_resume` в ответе:

<Tabs items={['Sync (JSON)', 'Stream (SSE)', 'Async']}>
  <Tab>
    Ответ включает полные данные паузы с URL resume:
    
    ```json
    {
      "success": true,
      "executionId": "<executionId>",
      "output": {
        "data": {
          "operation": "human",
          "_resume": {
            "apiUrl": "/api/resume/{workflowId}/{executionId}/{contextId}",
            "uiUrl": "/resume/{workflowId}/{executionId}",
            "contextId": "<contextId>",
            "executionId": "<executionId>",
            "workflowId": "<workflowId>"
          }
        }
      }
    }
    ```
  </Tab>
  <Tab>
    Блоки до HITL передают свои `selectedOutputs` в обычном режиме. Когда выполнение приостанавливается, финальное событие SSE включает `status: "paused"` и данные `_resume`:
    
    ```
    data: {"blockId":"agent1","chunk":"потоковый контент..."}
    data: {"event":"final","data":{"success":true,"output":{...,"_resume":{...}},"status":"paused"}}
    data: "[DONE]"
    ```
    
    При возобновлении блоки после HITL передают свои `selectedOutputs` таким же образом.
    
    <Callout type="info">
      Блоки HITL автоматически исключаются из выпадающего списка `selectedOutputs`, поскольку их данные всегда включены в ответ паузы.
    </Callout>
  </Tab>
  <Tab>
    Немедленно возвращает `202`. Используйте эндпоинт опроса для проверки, когда выполнение приостановится.
  </Tab>
</Tabs>

## Output'ы

| Output | Что это |
| --- | --- |
| `url` | URL портала подтверждения |
| `resumeEndpoint` | Эндпоинт API resume |
| `response` | Данные, показанные утверждающему |
| `submission` | Отправленная утверждающим форма |
| `submittedAt` | ISO-временная метка возобновления запуска |
| `<fieldName>` | Каждое поле формы Resume Form, по имени, после возобновления запуска |

Читайте их дальше как `<имяБлока.output>` — для блока с именем `approval` это `<approval.approved>`.

## Портал подтверждения

**Paused Output:**
```json
{
  "title": "<agent1.content.title>",
  "body": "<agent1.content.body>",
  "qualityScore": "<evaluator1.score>"
}
```

**Resume Input:**
```json
{
  "approved": { "type": "boolean" },
  "feedback": { "type": "string" }
}
```

**Использование дальше по цепочке:**
```javascript
// Блок Condition
<approval1.approved> === true
```
