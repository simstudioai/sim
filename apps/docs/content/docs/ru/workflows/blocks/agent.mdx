**Блок Agent** запускает модель внутри воркфлоу. Вы даёте ему инструкции, контекст и инструменты; модель рассуждает над входными данными, вызывает инструменты по необходимости и возвращает обычный текст или структурированный JSON, который последующие блоки читают по ссылке. Большинство воркфлоу строится вокруг одного или нескольких блоков Agent.

*Агент* и *блок Agent* — связанные, но разные понятия. Агент — это целый воркфлоу, который самостоятельно рассуждает и действует; блок Agent — это один шаг рассуждения внутри него. Простейший агент — это один блок Agent с инструментами, а более крупные объединяют несколько блоков с другими блоками. См. [Agents](/agents).

## Конфигурация

### Messages

Сообщения, отправляемые модели. Каждое сообщение имеет роль: **System** задаёт задачу и правила агента, **User** даёт ему входные данные для обработки. Вставьте [тег соединения](/workflows/connections), чтобы передать более ранний output, например `<start.input>`.

```markdown
Вы — ассистент поддержки для аналитического продукта.
Отвечайте в двух предложениях, указывайте использованный документ и никогда не угадывайте цену.
```

### Model

Модель, выполняющая шаг. По умолчанию `claude-sonnet-4-6`. Введите или выберите любую модель от OpenAI, Anthropic, Google, xAI, Groq, Cerebras, DeepSeek, Azure, AWS Bedrock, Google Vertex или OpenRouter, либо локальную модель через Ollama или VLLM.

### Files

Файлы для чтения моделью: изображения для модели с поддержкой зрения или документы для текста. Загрузите их в блоке или передайте файл от предыдущего блока, например, от триггера загрузки или ответа [API](/workflows/blocks/api), с помощью тега соединения.

### Tools

Возможности, которые агент может вызывать во время работы: поиск по базе знаний, отправка сообщения в Slack, запуск [Function](/workflows/blocks/function), вызов любой из [интеграций](/integrations) или использование [пользовательского инструмента](/agents/custom-tools) или [MCP-сервера](/agents/mcp), который вы добавили. Модель решает, какой инструмент вызывать и когда. (О том, откуда берутся инструменты и когда какой использовать, см. [Agents](/agents).) Каждый инструмент имеет управление использованием:

- **Auto.** Модель вызывает его, когда контекст этого требует.
- **Force.** Модель обязана вызывать его при каждом запуске.
- **None.** Инструмент скрыт от модели, что отключает его без удаления из блока.

### Skills

[Навыки агента](/agents/skills), которые агент может загружать по требованию: переиспользуемые пакеты инструкций, такие как стандарт кодирования или сценарий поддержки. Только имена навыков находятся в контексте изначально, а агент загружает полные инструкции, когда решает, что навык актуален.

### Memory

Встроенная память диалога, сохраняемая между запусками по идентификатору диалога:

- **None.** Каждый запуск независим.
- **Conversation.** Полная история для данного идентификатора диалога.
- **Sliding window (messages).** N последних сообщений.
- **Sliding window (tokens).** Последние сообщения в пределах бюджета токенов.

Памяти нужен идентификатор диалога для сохранения между запусками. Для памяти, общей между воркфлоу или управляемой как отдельное хранилище, используйте блок [Memory](/integrations/memory).

### Response Format

Задайте агенту JSON Schema для принудительного структурированного output'а. Ответ ограничивается схемой, и каждое поле становится собственным output'ом, читаемым по имени, например `<agent.sentiment>`. Без формата ответа агент возвращает обычный текст в `content`.

```json
{
  "name": "user_analysis",
  "schema": {
    "type": "object",
    "properties": {
      "sentiment": { "type": "string", "enum": ["positive", "negative", "neutral"] },
      "confidence": { "type": "number", "minimum": 0, "maximum": 1 }
    },
    "required": ["sentiment", "confidence"]
  }
}
```

### Advanced

Некоторые настройки находятся в расширенных или появляются только для моделей, которые их поддерживают:

- **Temperature.** Насколько разнообразен output. Держите низким (0–0,3), когда нужна точность и повторяемость, средним (около 0,5) для повседневной работы, выше (0,7+) для творческого разнообразия.
- **Max output tokens.** Ограничивает длину ответа. По умолчанию — полный лимит модели.
- **Reasoning effort / Thinking level.** Для моделей с расширенным рассуждением — сколько модель думает перед ответом. Выше — тщательнее, но медленнее и дороже по токенам.
- **API key.** Ваш ключ для выбранного провайдера. Скрыт на хостинге Sim, который предоставляет свой.

## Output'ы

После выполнения агента последующие блоки читают его результат по имени:

| Output | Что это |
| --- | --- |
| `<agent.content>` | Ответ: текст или структурированный объект, если задан формат ответа |
| `<agent.tokens>` | Использование токенов, объект `{ input, output, total }` |
| `<agent.toolCalls>` | Вызванные агентом инструменты с их входными данными и результатами |
| `<agent.model>` | Модель, выполнившая шаг |
| `<agent.cost>` | Оценка стоимости вызова |

Когда задан формат ответа, его поля доступны напрямую, например `<agent.sentiment>`.

## Рекомендации

- **Пишите чёткое системное сообщение.** Определите роль, тон и ограничения агента. Конкретные инструкции дают более надёжный результат, чем расплывчатый промпт.
- **Подбирайте модель и temperature под задачу.** Используйте более сильную модель и низкую temperature (0–0,3) для точности; повышайте temperature для творческого или разнообразного output'а.
- **Давайте агенту только нужные инструменты.** Слишком много инструментов размывают выбор. Для задач, которые не пересекаются, используйте второй блок Agent вместо перегрузки одного.
- **Используйте формат ответа, когда последующему блоку нужны конкретные поля.** Это гарантирует структуру, и вы читаете каждое поле как `<agent.поле>`.
