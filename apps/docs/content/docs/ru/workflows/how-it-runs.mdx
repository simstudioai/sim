Когда вы запускаете воркфлоу, Sim определяет порядок из [соединений](/workflows/connections): блок выполняется, как только блоки, от которых он зависит, завершились.

## Блоки выполняются, как только могут

Несколько блоков в воркфлоу могут выполняться одновременно. Блок запускается в тот момент, когда его зависимости завершаются, и не ждёт ничего другого.

<Image src="/static/execution/concurrency.png" alt="Два блока Agent выполняются после триггера Start" width={800} height={500} />

Здесь агенты Customer Support и Deep Researcher каждый зависят только от Start, поэтому ни один не ждёт другого.

## Блок ждёт все свои входные данные

Когда несколько блоков ведут в один, этот блок ждёт каждого источника, который будет выполнен, и затем запускается один раз, имея доступ ко всем их output'ам. Источник на ветке, которая не была выбрана, не задерживает его. Вам не нужно объединять output'ы самостоятельно.

<Image src="/static/execution/combination.png" alt="Блок Function выполняется после завершения двух блоков Agent" width={800} height={500} />

Здесь блок Function выполняется после завершения обоих агентов, имея готовыми оба их output'а.

## Ветвления следуют по одному пути

Воркфлоу может разделяться. Блок [Condition](/workflows/blocks/condition) ветвится по явному правилу; блок [Router](/workflows/blocks/router) позволяет модели выбрать путь. Выполняется только выбранная ветка. Блок на ветке, которая не была выполнена, не производит output, поэтому [тег соединения](/workflows/connections), указывающий на него, возвращает пустое значение.

<Image src="/static/execution/routing.png" alt="Воркфлоу, ветвящийся по условию и по маршрутизатору" width={800} height={500} />

Для повторения работы блок [Loop](/workflows/blocks/loop) выполняет внутренние блоки по списку, счётчику или пока выполняется условие, а блок [Parallel](/workflows/blocks/parallel) выполняет их для нескольких элементов одновременно.

<Callout type="info">
Воркфлоу может вызывать другие воркфлоу через блок Workflow, инструмент MCP или блок API. Sim отслеживает цепочку вызовов и останавливает запуск, превышающий 25 переходов, чтобы воркфлоу, вызывающие друг друга, не могли зациклиться навсегда.
</Callout>

## Когда блок завершается с ошибкой

Блок с ошибкой прерывает запуск: уже выполняющиеся блоки завершаются, новые не запускаются. Чтобы вместо этого обработать ошибку, подключите **порт ошибки** блока — выполнение пойдёт по [пути ошибки](/workflows/connections) и продолжится.

<Image src="/static/execution/error-path.png" alt="Блок Function, чей порт ошибки ведёт к блоку-обработчику, а обычный путь продолжается к блоку успеха" width={800} height={319} />

Здесь `throwError` завершается с ошибкой, поэтому выполнение выходит через его красный порт ошибки к `handleError`; `handleSuccess` на обычном пути никогда не выполняется.

## Сколько может длиться запуск

Синхронный запуск ограничен **5 минутами на бесплатном тарифе** и **50 минутами на платных тарифах**; асинхронный запуск получает **90 минут**. Запуск, достигший предела, завершается с ошибкой «Execution timed out». Обычные причины — цикл по большому списку или медленный вызов агента или API — разделите работу или запустите её асинхронно. При самостоятельном хостинге эти ограничения можно настроить.

## Наблюдение за запуском

Пока воркфлоу выполняется, редактор показывает состояние каждого блока в реальном времени: в очереди, выполняется, завершён или ошибка. Каждый запуск записывается, поэтому вы можете открыть [логи](/logs-debugging), чтобы увидеть, что каждый блок получил и вернул после завершения.
